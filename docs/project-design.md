# RAG SDK 项目设计

## 文档定位

本文档描述 RAG SDK 的目标架构与总体职责边界。

本文档只定义项目级设计，不展开具体接口、类型、流程细节。每个子包的详细设计由独立文档维护。

## 项目定位

RAG SDK 面向 RAG 应用开发中的通用工程链路，提供一组稳定、清晰、可组合的基础协议与流程编排能力。

SDK 的核心目标不是替业务系统完成全部 RAG 能力，而是把 RAG 链路中容易重复定义、重复衔接、重复约定的部分沉淀为统一边界，使不同来源的文档、分块、向量、索引、检索能力可以在一致的数据结构和流程语义下协作。

## 设计原则

1. SDK 只定义当前职责所需的最小能力。
2. 核心包保持稳定，具体能力通过子包或外部实现接入。
3. 协议优先于具体实现。
4. 业务逻辑不进入 SDK。
5. 外部能力适配保持边界清晰，不把第三方框架模型泄漏到核心协议。
6. 不为了未来可能出现的需求提前引入复杂抽象。

## 总体架构

目标架构按职责分为以下层次：

```text
应用层
  └─ 使用 SDK 组织业务侧 RAG 流程

RAG SDK
  ├─ @rag-sdk/core
  │    └─ 核心数据结构、基础类型、通用协议
  ├─ @rag-sdk/indexing
  │    └─ 索引阶段协议与索引流程编排
  ├─ @rag-sdk/retrieval
  │    └─ 检索阶段协议与检索流程编排
  ├─ @rag-sdk/adapters
  │    └─ 外部能力到 SDK 协议的适配
  └─ @rag-sdk/utils
       └─ 跨包复用的内部工具

外部系统
  ├─ 文档来源
  ├─ 文档解析、切分、Embedding、Rerank 等能力
  ├─ 向量数据库或其他索引存储
  └─ 业务系统、权限系统、应用服务、UI
```

## SDK 负责什么

SDK 负责以下内容：

1. 定义 RAG 链路中的核心数据结构。
2. 定义索引、检索等流程中的阶段协议。
3. 提供基础流程编排能力。
4. 统一不同阶段之间的数据传递语义。
5. 提供必要的外部能力适配入口。
6. 为业务侧接入不同实现提供清晰边界。

## SDK 不负责什么

SDK 不负责以下内容：

1. 不提供模型服务本身。
2. 不内置具体大模型、Embedding、Rerank、OCR、解析器等能力。
3. 不负责具体向量存储能力适配。
4. 不负责数据库部署、迁移、运维或性能调优。
5. 不承载业务流程、业务规则、权限模型、租户模型。
6. 不提供应用服务、HTTP API、后台任务系统或 UI。
7. 不替业务系统决定数据生命周期、索引刷新策略或权限过滤策略。

其中，向量存储属于外部系统能力。SDK 可以在流程边界上表达索引结果如何交给外部系统，但不提供具体向量数据库适配实现，也不把某个向量数据库作为架构前提。

## 子包职责概览

### @rag-sdk/core

`core` 是 SDK 的基础层，负责定义跨子包共享的核心数据结构、基础类型和基础协议。

它不依赖其他 SDK 子包，也不包含具体业务流程。

详细设计见 [core](./packages/core.md)。

### @rag-sdk/indexing

`indexing` 负责定义索引链路中的阶段协议与流程编排。

它描述从文档加载、转换、切分、向量化到索引结果输出之间的协作关系，但不实现具体 loader、chunker、embedder、metadata extractor 或向量存储适配。

详细设计见 [indexing](./packages/indexing.md)。

### @rag-sdk/retrieval

`retrieval` 负责定义检索链路中的阶段协议与流程编排。

它描述查询输入、召回、重排、过滤、结果组装等阶段的协作边界，但不实现具体搜索引擎、向量数据库、rerank 模型或业务权限逻辑。

详细设计见 [retrieval](./packages/retrieval.md)。

### @rag-sdk/adapters

`adapters` 负责把外部生态能力转换为 SDK 可以消费的协议实现。

它只做边界适配，不承载 SDK 核心协议定义，也不把第三方框架类型扩散到 `core`、`indexing`、`retrieval`。

详细设计见 [adapters](./packages/adapters.md)。

### @rag-sdk/utils

`utils` 负责提供跨子包复用的内部通用工具。

它不承载业务语义，也不定义核心流程协议。

详细设计见 [utils](./packages/utils.md)。

### @rag-sdk/vector-stores

目标架构中，SDK 不负责具体向量存储能力适配。

因此 `vector-stores` 不作为目标架构中的核心对外能力展开。如果仓库中保留该子包，它的设计文档只用于说明该边界，不继续扩展具体向量数据库适配职责。

详细说明见 [vector-stores](./packages/vector-stores.md)。

## 包依赖方向

目标依赖方向如下：

```text
core
  ↑
  ├─ indexing
  ├─ retrieval
  └─ adapters

utils
  ↑
  ├─ adapters
  └─ 其他确有复用需要的子包
```

约束：

1. `core` 不依赖其他 SDK 子包。
2. `indexing` 和 `retrieval` 可以依赖 `core`。
3. `adapters` 可以依赖具体需要适配的协议包。
4. `utils` 只放无业务语义的公共工具。
5. 子包之间不得形成循环依赖。

## 设计边界

SDK 的设计边界是协议和流程，不是完整 RAG 平台。

业务系统仍然负责决定：

1. 数据从哪里来。
2. 数据如何授权。
3. 数据何时索引、更新、删除。
4. 使用哪个模型和外部服务。
5. 使用哪个数据库或索引系统。
6. 检索结果如何进入最终业务响应。

SDK 只让这些能力可以在一致的协议下被组织和衔接。
